# Pester 5 Best Practices and Standards

## Running Pester Tests — Always Use Start-Process (Detached)

> **CRITICAL**: Never run Pester tests directly inside the VS Code integrated terminal —
> not even via `pwsh -NoProfile -Command { ... }`. Both in-process invocations **and**
> synchronous child-process calls hang VS Code because the terminal waits for the child
> to exit and Pester/module output can stall VS Code's terminal renderer or the
> PowerShell Extension Host pipe.
>
> **Always use `Start-Process` (fully detached)** — it creates a process with no
> parent-child pipe, so VS Code's terminal never blocks on it.

### Recommended: Start-Process with Log Polling

```powershell
# 1. Start tests detached — log goes to output/ (gitignored, never project root)
$logPath = Join-Path $PWD 'output\test.log'
New-Item -Path (Split-Path $logPath) -ItemType Directory -Force | Out-Null
Remove-Item $logPath -ErrorAction SilentlyContinue

Start-Process -FilePath pwsh -ArgumentList @(
    '-NoProfile', '-NonInteractive', '-Command',
    "Set-Location '$PWD'; Invoke-Pester -Path './tests' -Output Detailed *>&1 | Out-File -FilePath '$logPath' -Encoding utf8"
) -WindowStyle Hidden -PassThru

# 2. Poll for completion
for ($i = 0; $i -lt 60; $i++) {
    Start-Sleep 3
    if (Test-Path $logPath) {
        $c = Get-Content $logPath -Raw -ErrorAction SilentlyContinue
        if ($c -match 'Tests Passed|Tests Failed|Invoke-Pester.*completed') {
            Get-Content $logPath -Tail 30
            break
        }
    }
}
```

### Sampler Projects (also detached)

```powershell
$logPath = Join-Path $PWD 'output\test.log'
New-Item -Path (Split-Path $logPath) -ItemType Directory -Force | Out-Null
Remove-Item $logPath -ErrorAction SilentlyContinue

Start-Process -FilePath pwsh -ArgumentList @(
    '-NoProfile', '-NonInteractive', '-Command',
    "Set-Location '$PWD'; .\build.ps1 -Tasks test *>&1 | Out-File -FilePath '$logPath' -Encoding utf8"
) -WindowStyle Hidden -PassThru

# Poll for completion
for ($i = 0; $i -lt 60; $i++) {
    Start-Sleep 3
    if (Test-Path $logPath) {
        $c = Get-Content $logPath -Raw -ErrorAction SilentlyContinue
        if ($c -match 'Build (FAILED|succeeded)') {
            Get-Content $logPath -Tail 30
            break
        }
    }
}
```

### Why Start-Process (not pwsh -Command)?

| Problem | Cause | Solution |
|---|---|---|
| VS Code hangs with `pwsh -Command` | Terminal synchronously waits for child process; Pester output stalls the pipe | `Start-Process` (detached) — no parent-child pipe |
| VS Code hangs with in-process Pester | Blocks the PowerShell extension thread | `Start-Process` (detached) |
| Module state leaks between test runs | `Import-Module -Force` in-process may not fully reload assemblies | A new process starts with a clean module cache |
| `InModuleScope` deadlocks | Locking conflicts with the language server | Isolated process avoids shared locks |
| Stale breakpoints or $Error state | Previous debug sessions pollute the session | `-NoProfile` guarantees a clean session |

### Key Rules

- **Log files go to `output/`** (gitignored), never the project root.
- **Always use `-WindowStyle Hidden`** to prevent a visible console flash.
- Poll the log for completion markers (`Tests Passed`, `Build succeeded`, etc.).
- For Sampler projects, skip `-ResolveDependency` if `output/RequiredModules/` already exists.

### Terminal Commands Generated by Copilot

When generating terminal commands to run Pester tests, **always** use `Start-Process`:

```powershell
# WRONG — runs in-process, hangs VS Code
Invoke-Pester -Path './tests'

# WRONG — synchronous child process, still hangs VS Code
pwsh -NoProfile -NonInteractive -Command { Invoke-Pester -Path './tests' -Output Detailed }

# CORRECT — fully detached, VS Code never blocks
Start-Process -FilePath pwsh -ArgumentList @(
    '-NoProfile', '-NonInteractive', '-Command',
    "Set-Location '$PWD'; Invoke-Pester -Path './tests' -Output Detailed *>&1 | Out-File -FilePath 'output\test.log' -Encoding utf8"
) -WindowStyle Hidden -PassThru
```

## Pester 5 Fundamentals

### Version Requirements

- **ALWAYS** write tests for Pester 5.x (5.3+)
- **NEVER** use Pester 4 syntax (`Should Be`, `Should Throw`, `Assert-MockCalled`)
- If migrating from Pester 4, convert all assertions to the Pester 5 `Should -<Operator>` syntax

### Discovery vs Run Phase

Pester 5 has two distinct phases:

- **Discovery phase**: Pester scans all `Describe`, `Context`, and `It` blocks to build a test plan. Code outside `BeforeAll`, `BeforeEach`, `AfterAll`, `AfterEach`, and `It` blocks runs during discovery.
- **Run phase**: Pester executes the tests according to the plan.

**Critical rule**: Never put executable code directly inside `Describe` or `Context` blocks. It will run during discovery, not during test execution.

```powershell
# WRONG — runs during discovery, not during test execution
Describe 'Get-Widget' {
    $widget = Get-Widget -Name 'Test'  # Runs at discovery time!

    It 'Should return a widget' {
        $widget | Should -Not -BeNullOrEmpty
    }
}

# CORRECT — runs during the run phase
Describe 'Get-Widget' {
    BeforeAll {
        $widget = Get-Widget -Name 'Test'
    }

    It 'Should return a widget' {
        $widget | Should -Not -BeNullOrEmpty
    }
}
```

### BeforeAll vs BeforeEach

| Block | Scope | When to Use |
|---|---|---|
| `BeforeAll` | Runs once before all tests in the current `Describe`/`Context` | Expensive setup: importing modules, creating test fixtures, one-time mocks |
| `BeforeEach` | Runs before every `It` block | Per-test setup: resetting variables, creating fresh objects, per-test mocks |
| `AfterAll` | Runs once after all tests | Cleanup: removing temp files, restoring state |
| `AfterEach` | Runs after every `It` block | Per-test cleanup: resetting mocks, removing per-test artifacts |

```powershell
Describe 'File operations' {
    BeforeAll {
        # Runs once — import the module under test
        $modulePath = Join-Path $PSScriptRoot '..' '..' 'output' 'builtModule' 'MyModule'
        Import-Module $modulePath -Force
    }

    BeforeEach {
        # Runs before each It — create a fresh temp file
        $script:tempFile = New-TemporaryFile
    }

    AfterEach {
        # Runs after each It — clean up
        Remove-Item $script:tempFile -ErrorAction SilentlyContinue
    }

    AfterAll {
        # Runs once — unload the module
        Remove-Module MyModule -ErrorAction SilentlyContinue
    }

    It 'Should write content' {
        Set-Content -Path $script:tempFile -Value 'test'
        Get-Content $script:tempFile | Should -Be 'test'
    }

    It 'Should overwrite content' {
        Set-Content -Path $script:tempFile -Value 'original'
        Set-Content -Path $script:tempFile -Value 'updated'
        Get-Content $script:tempFile | Should -Be 'updated'
    }
}
```

### Variable Scoping

In Pester 5, variables set in `BeforeAll` or `BeforeEach` must use `$script:` scope or the `$testVariable` pattern to be accessible in `It` blocks.

```powershell
Describe 'Scoping example' {
    BeforeAll {
        # Use script scope to share across It blocks
        $script:config = @{ Key = 'Value' }
    }

    It 'Should access the config' {
        $script:config.Key | Should -Be 'Value'
    }
}
```

**Exception**: Variables set in `BeforeEach` within the same `Describe`/`Context` are available in `It` blocks without `$script:` scope because Pester 5 runs `BeforeEach` and `It` in the same scope.

## File and Naming Conventions

### File Naming

- Test files **must** use the suffix `.Tests.ps1` (capital T)
- Name test files to match the function or script under test:
  - Function `Get-Widget` → `Get-Widget.Tests.ps1`
  - Script `Deploy-App.ps1` → `Deploy-App.Tests.ps1`
  - Module `MyModule.psm1` → `MyModule.Tests.ps1` (for module-level tests)

### Directory Structure (Sampler Convention)

```
tests/
├── QA/
│   └── module.tests.ps1       # Quality assurance tests (exported commands, help, aliases)
├── Unit/
│   ├── Public/
│   │   ├── Get-Widget.Tests.ps1
│   │   └── Set-Widget.Tests.ps1
│   └── Private/
│       └── ConvertTo-InternalFormat.Tests.ps1
└── Integration/
    └── End-to-End.Tests.ps1
```

### Encoding

- Use **UTF-8 with BOM** for test files (PowerShell convention)
- Use CRLF line endings on Windows, LF on Linux/macOS

## Describe and Context Structure

### Describe Block

- **One `Describe` block per function or logical unit**
- Use the function name as the `Describe` name
- Do not nest `Describe` blocks (use `Context` for sub-grouping)

```powershell
Describe 'Get-Widget' {
    # Tests for Get-Widget
}
```

### Context Block

- Use `Context` to group tests by scenario, parameter set, or condition
- Use descriptive names that read like sentences when combined with `When` or `Given`

```powershell
Describe 'Get-Widget' {
    Context 'When called with a valid name' {
        It 'Should return the widget' {
            # ...
        }
    }

    Context 'When the widget does not exist' {
        It 'Should return $null' {
            # ...
        }

        It 'Should not throw an error' {
            # ...
        }
    }

    Context 'When called with pipeline input' {
        It 'Should accept pipeline input' {
            # ...
        }
    }
}
```

### It Block (Test Cases)

- Start `It` descriptions with `Should` (preferred) or a descriptive action
- Each `It` block should test **one thing** (single assertion or closely related assertions)
- Keep `It` descriptions concise but descriptive enough to diagnose failures without reading the code

```powershell
# Good
It 'Should return a non-empty string' { }
It 'Should throw when Name is null' { }
It 'Should call Get-Item exactly once' { }

# Avoid
It 'Test 1' { }
It 'Works' { }
It 'Should return the correct value and not throw and write verbose output' { }
```

## Assertions

### Pester 5 Assertion Syntax

**ALWAYS** use the Pester 5 `Should -<Operator>` syntax. **NEVER** use Pester 4 `Should Be` syntax.

```powershell
# Pester 5 (CORRECT)
$result | Should -Be 'expected'
$result | Should -BeExactly 'Expected'
$result | Should -Not -BeNullOrEmpty
$result | Should -HaveCount 3
$result | Should -Contain 'item'
$result | Should -BeOfType [string]
$result | Should -Match '^prefix-\d+$'
$result | Should -BeLike 'prefix-*'
$result | Should -BeGreaterThan 0
$result | Should -BeTrue
$result | Should -BeFalse
$result | Should -BeIn @('a', 'b', 'c')

# Pester 4 (WRONG — do NOT use)
$result | Should Be 'expected'
$result | Should Not BeNullOrEmpty
```

### Common Assertions Reference

| Operator | Purpose | Example |
|---|---|---|
| `-Be` | Equality (case-insensitive) | `$x \| Should -Be 'hello'` |
| `-BeExactly` | Equality (case-sensitive) | `$x \| Should -BeExactly 'Hello'` |
| `-BeNullOrEmpty` | Null or empty check | `$x \| Should -Not -BeNullOrEmpty` |
| `-BeTrue` / `-BeFalse` | Boolean check | `$flag \| Should -BeTrue` |
| `-BeOfType` | Type check | `$x \| Should -BeOfType [string]` |
| `-BeGreaterThan` | Numeric comparison | `$count \| Should -BeGreaterThan 0` |
| `-BeLessThan` | Numeric comparison | `$count \| Should -BeLessThan 100` |
| `-BeIn` | Value in collection | `$x \| Should -BeIn @(1, 2, 3)` |
| `-Contain` | Collection contains value | `$list \| Should -Contain 'item'` |
| `-HaveCount` | Collection length | `$list \| Should -HaveCount 5` |
| `-Match` | Regex match | `$x \| Should -Match '^\d+$'` |
| `-BeLike` | Wildcard match | `$x \| Should -BeLike 'prefix-*'` |
| `-Throw` | Exception check | `{ Get-Foo } \| Should -Throw` |
| `-Not` | Negation modifier | `$x \| Should -Not -Be $null` |

### Testing for Exceptions

```powershell
# Test that a function throws
It 'Should throw when Name is null' {
    { Get-Widget -Name $null } | Should -Throw
}

# Test for specific exception type
It 'Should throw ArgumentException' {
    { Get-Widget -Name '' } | Should -Throw -ExceptionType ([System.ArgumentException])
}

# Test for specific error message
It 'Should throw with message about missing name' {
    { Get-Widget -Name '' } | Should -Throw -ExpectedMessage '*name*'
}

# Test that it does NOT throw
It 'Should not throw with valid input' {
    { Get-Widget -Name 'Valid' } | Should -Not -Throw
}
```

### Multiline and Complex Assertions

```powershell
# Test multiline output — use (?s) for dotall mode
It 'Should return base64-wrapped content' {
    $result | Should -Match '(?s)^\[ENC=.*\]$'
}

# Test multiple properties of an object
It 'Should return a widget with correct properties' {
    $result = Get-Widget -Name 'Test'
    $result.Name | Should -Be 'Test'
    $result.Status | Should -Be 'Active'
    $result.CreatedDate | Should -BeOfType [datetime]
}
```

## Mocking

### Basic Mocking

```powershell
Describe 'Send-Report' {
    BeforeAll {
        Mock -CommandName Send-MailMessage
    }

    It 'Should send an email' {
        Send-Report -To 'user@example.com' -Subject 'Test'
        Should -Invoke -CommandName Send-MailMessage -Times 1 -Exactly
    }
}
```

### Module-Scoped Mocks

When the function under test is in a module, the mock must target the module's scope. Otherwise the module resolves the real command, not your mock.

```powershell
Describe 'Get-Widget' {
    BeforeAll {
        Mock -ModuleName MyModule -CommandName Get-Item -MockWith {
            [PSCustomObject]@{ Name = 'MockedItem' }
        }
    }

    It 'Should use the mocked Get-Item' {
        $result = Get-Widget -Name 'Test'
        $result.Name | Should -Be 'MockedItem'
        Should -Invoke -ModuleName MyModule -CommandName Get-Item -Times 1
    }
}
```

**Critical**: Always match `-ModuleName` between `Mock` and `Should -Invoke`.

### Mock with Parameter Filters

```powershell
Mock -CommandName Get-Item -ParameterFilter { $Path -eq 'C:\important' } -MockWith {
    [PSCustomObject]@{ Name = 'ImportantFile' }
}

Mock -CommandName Get-Item -ParameterFilter { $Path -eq 'C:\other' } -MockWith {
    [PSCustomObject]@{ Name = 'OtherFile' }
}
```

### Mock with Return Values

```powershell
# Return a specific object
Mock -CommandName Get-Service -MockWith {
    [PSCustomObject]@{
        Name   = 'TestService'
        Status = 'Running'
    }
}

# Return nothing (void)
Mock -CommandName Stop-Service

# Throw from a mock
Mock -CommandName Get-Item -MockWith {
    throw [System.IO.FileNotFoundException]::new('File not found')
}
```

### RemoveParameterValidation

When mocking commands with `[ValidateScript()]` attributes that call internal functions, use `-RemoveParameterValidation`:

```powershell
# The real Protect-Data has [ValidateScript()] that calls Get-ProtectedDataSupportedType
# which is not exported. The mock would fail without removing the validation.
Mock -ModuleName MyModule -CommandName Protect-Data `
    -RemoveParameterValidation 'InputObject' `
    -MockWith { 'ProtectedBlob' }
```

### Should -Invoke

```powershell
# Verify a mock was called
Should -Invoke -CommandName Get-Item -Times 1 -Exactly

# Verify with parameter filter
Should -Invoke -CommandName Get-Item -Times 1 -ParameterFilter { $Path -eq 'C:\test' }

# Verify a mock was NOT called
Should -Invoke -CommandName Remove-Item -Times 0

# Verify at least N calls
Should -Invoke -CommandName Write-Verbose -Times 2 -Exactly:$false
```

**Pester 4 migration note**: Replace `Assert-MockCalled` with `Should -Invoke`. The parameters are the same.

## Test Data and Fixtures

### TestDrive

Pester provides `TestDrive:` — a temporary PSDrive that is automatically cleaned up after each `Describe` block.

```powershell
Describe 'Export-Config' {
    BeforeAll {
        $testFile = Join-Path TestDrive: 'config.json'
    }

    It 'Should create a config file' {
        Export-Config -Path $testFile -Settings @{ Key = 'Value' }
        Test-Path $testFile | Should -BeTrue
    }

    It 'Should write valid JSON' {
        Export-Config -Path $testFile -Settings @{ Key = 'Value' }
        $content = Get-Content $testFile -Raw | ConvertFrom-Json
        $content.Key | Should -Be 'Value'
    }
}
```

### TestRegistry

Pester provides `TestRegistry:` for registry-based tests (Windows only).

```powershell
Describe 'Set-AppSetting' {
    BeforeAll {
        $testKey = Join-Path TestRegistry: 'MyApp'
        New-Item -Path $testKey -Force
    }

    It 'Should write a registry value' {
        Set-AppSetting -Key $testKey -Name 'Setting1' -Value 'Test'
        Get-ItemPropertyValue -Path $testKey -Name 'Setting1' | Should -Be 'Test'
    }
}
```

### In-Memory Test Data

Prefer in-memory test data over file-based fixtures when possible:

```powershell
BeforeAll {
    $testUsers = @(
        [PSCustomObject]@{ Name = 'Alice'; Department = 'Engineering' }
        [PSCustomObject]@{ Name = 'Bob'; Department = 'Marketing' }
        [PSCustomObject]@{ Name = 'Charlie'; Department = 'Engineering' }
    )
}
```

## Tagging Tests

### Tag Convention

Use `-Tag` on `Describe`, `Context`, or `It` blocks to categorize tests:

```powershell
Describe 'Get-Widget' -Tag 'Unit' {
    Context 'Basic functionality' -Tag 'Smoke' {
        It 'Should return a widget' -Tag 'Critical' {
            # ...
        }
    }
}
```

### Standard Tags

| Tag | Purpose |
|---|---|
| `Unit` | Unit tests (isolated, no external dependencies) |
| `Integration` | Integration tests (touches real resources) |
| `Acceptance` | End-to-end acceptance tests |
| `Smoke` | Quick smoke tests for CI/CD |
| `Slow` | Long-running tests |
| `WindowsOnly` | Tests that require Windows |
| `LinuxOnly` | Tests that require Linux |
| `RequiresAdmin` | Tests that need elevated privileges |

### Running Tagged Tests

```powershell
# Run only unit tests
Invoke-Pester -Tag 'Unit'

# Exclude slow tests
Invoke-Pester -ExcludeTag 'Slow'

# Run smoke tests only
Invoke-Pester -Tag 'Smoke'
```

## Pester Configuration

### Configuration Object (Pester 5)

```powershell
$config = New-PesterConfiguration
$config.Run.Path = './tests'
$config.Run.Exit = $true
$config.CodeCoverage.Enabled = $true
$config.CodeCoverage.Path = @('./source/Public/*.ps1', './source/Private/*.ps1')
$config.CodeCoverage.OutputFormat = 'JaCoCo'
$config.CodeCoverage.OutputPath = './output/coverage.xml'
$config.TestResult.Enabled = $true
$config.TestResult.OutputFormat = 'NUnitXml'
$config.TestResult.OutputPath = './output/testResults.xml'
$config.Output.Verbosity = 'Detailed'
$config.Filter.Tag = @('Unit')
$config.Filter.ExcludeTag = @('Slow')

Invoke-Pester -Configuration $config
```

### Sampler Integration

In Sampler projects, Pester configuration is managed through `build.yaml`:

```yaml
Pester:
  Configuration:
    Run:
      Path:
        - tests/QA
        - tests/Unit
    Output:
      Verbosity: Detailed
    CodeCoverage:
      Enabled: true
      OutputFormat: JaCoCo
      CoveragePercentTarget: 80
    TestResult:
      Enabled: true
      OutputFormat: NUnitXml
```

Do not duplicate Pester configuration in test files when using Sampler — let `build.yaml` handle it.

## Quality Assurance Tests (Sampler Pattern)

### Module Quality Tests

The standard QA test file validates module structure and help:

```powershell
Describe 'Module Quality Tests' -Tag 'QA' {
    BeforeAll {
        $moduleName = 'MyModule'
        $modulePath = Join-Path $PSScriptRoot '..' '..' 'output' 'builtModule' $moduleName
        Import-Module $modulePath -Force -ErrorAction Stop
        $module = Get-Module $moduleName
    }

    Context 'Module manifest' {
        It 'Should have a valid module manifest' {
            Test-ModuleManifest -Path "$modulePath\$moduleName.psd1" | Should -Not -BeNullOrEmpty
        }

        It 'Should have a non-empty GUID' {
            $module.Guid | Should -Not -Be ([guid]::Empty)
        }

        It 'Should have a version' {
            $module.Version | Should -Not -BeNullOrEmpty
        }
    }

    Context 'Exported commands' {
        BeforeAll {
            $exportedCommands = $module.ExportedCommands.Keys
        }

        It 'Should export at least one command' {
            $exportedCommands | Should -Not -BeNullOrEmpty
        }

        It 'Should use approved verbs for all exported functions' {
            $approvedVerbs = (Get-Verb).Verb
            $exportedCommands | ForEach-Object {
                $verb = ($_ -split '-')[0]
                $verb | Should -BeIn $approvedVerbs -Because "$_ uses an unapproved verb '$verb'"
            }
        }
    }

    Context 'Help content' {
        BeforeAll {
            $commands = Get-Command -Module $moduleName
        }

        It 'Should have help for <_.Name>' -ForEach ($commands) {
            $help = Get-Help $_.Name -Full
            $help.Synopsis | Should -Not -BeNullOrEmpty
            $help.Description | Should -Not -BeNullOrEmpty
        }

        It 'Should have examples for <_.Name>' -ForEach ($commands) {
            $help = Get-Help $_.Name -Full
            $help.Examples.Example | Should -Not -BeNullOrEmpty
        }
    }
}
```

## Data-Driven Tests (ForEach / TestCases)

### Pester 5 ForEach Syntax

```powershell
Describe 'ConvertTo-Celsius' {
    It 'Should convert <Fahrenheit>°F to <Expected>°C' -ForEach @(
        @{ Fahrenheit = 32;  Expected = 0 }
        @{ Fahrenheit = 212; Expected = 100 }
        @{ Fahrenheit = 72;  Expected = 22.22 }
    ) {
        ConvertTo-Celsius -Fahrenheit $Fahrenheit | Should -Be $Expected
    }
}
```

### ForEach on Context

```powershell
Describe 'Get-Widget' {
    Context 'When called with ParameterSet <SetName>' -ForEach @(
        @{ SetName = 'ByName'; Params = @{ Name = 'Test' } }
        @{ SetName = 'ById';   Params = @{ Id = 1 } }
    ) {
        It 'Should return a widget' {
            $result = Get-Widget @Params
            $result | Should -Not -BeNullOrEmpty
        }
    }
}
```

**Pester 4 migration note**: Replace `-TestCases` with `-ForEach`. The hashtable format is the same.

## Testing Private Functions

### Access via InModuleScope

```powershell
Describe 'ConvertTo-InternalFormat' {
    It 'Should convert the input to internal format' {
        InModuleScope MyModule {
            $result = ConvertTo-InternalFormat -Input 'test'
            $result | Should -Be 'TEST_INTERNAL'
        }
    }
}
```

### Pass Parameters to InModuleScope

```powershell
Describe 'ConvertTo-InternalFormat' {
    It 'Should convert <Input> to <Expected>' -ForEach @(
        @{ Input = 'test'; Expected = 'TEST_INTERNAL' }
    ) {
        InModuleScope MyModule -Parameters @{ Input = $Input; Expected = $Expected } {
            $result = ConvertTo-InternalFormat -Input $Input
            $result | Should -Be $Expected
        }
    }
}
```

**Guideline**: Prefer testing private functions indirectly through public functions. Use `InModuleScope` only when direct testing is necessary.

## Common Patterns and Anti-Patterns

### Pattern: Arrange-Act-Assert

```powershell
It 'Should process the widget correctly' {
    # Arrange
    $widget = [PSCustomObject]@{ Name = 'Test'; Status = 'Pending' }

    # Act
    $result = Invoke-WidgetProcessor -Widget $widget

    # Assert
    $result.Status | Should -Be 'Processed'
}
```

### Anti-Pattern: Multiple Unrelated Assertions

```powershell
# BAD — testing too many things in one It block
It 'Should work correctly' {
    Get-Widget -Name 'A' | Should -Not -BeNullOrEmpty
    Get-Widget -Name 'B' | Should -Not -BeNullOrEmpty
    Remove-Widget -Name 'A' | Should -Not -Throw
    Get-Widget | Should -HaveCount 1
}

# GOOD — one concept per It block
It 'Should return widget A' {
    Get-Widget -Name 'A' | Should -Not -BeNullOrEmpty
}

It 'Should return widget B' {
    Get-Widget -Name 'B' | Should -Not -BeNullOrEmpty
}
```

### Anti-Pattern: Hardcoded Paths

```powershell
# BAD
$config = Get-Content 'C:\Users\dev\project\config.json'

# GOOD — use TestDrive or $PSScriptRoot
$config = Get-Content (Join-Path TestDrive: 'config.json')
$config = Get-Content (Join-Path $PSScriptRoot 'fixtures' 'config.json')
```

### Anti-Pattern: Test Interdependence

```powershell
# BAD — test 2 depends on test 1 creating the file
It 'Should create a file' {
    New-Item -Path TestDrive:\data.txt -Value 'test'
    Test-Path TestDrive:\data.txt | Should -BeTrue
}

It 'Should read the file' {
    Get-Content TestDrive:\data.txt | Should -Be 'test'  # Depends on test above!
}

# GOOD — each test sets up its own state
It 'Should read a file' {
    New-Item -Path TestDrive:\data.txt -Value 'test' -Force
    Get-Content TestDrive:\data.txt | Should -Be 'test'
}
```

### Anti-Pattern: Not Cleaning Up Mocks

Pester 5 automatically scopes mocks to the block they are defined in. However, if you define mocks in `BeforeAll` at the `Describe` level, they persist for the entire `Describe`. Use `Context` blocks with their own `BeforeAll` to scope mocks more narrowly.

## Code Coverage

### Enabling Code Coverage

```powershell
$config = New-PesterConfiguration
$config.CodeCoverage.Enabled = $true
$config.CodeCoverage.Path = @('./source/Public/*.ps1')
$config.CodeCoverage.OutputFormat = 'JaCoCo'
$config.CodeCoverage.OutputPath = './output/coverage.xml'
$config.CodeCoverage.CoveragePercentTarget = 80

Invoke-Pester -Configuration $config
```

### Coverage Best Practices

- Set a reasonable coverage target (70-80% for most projects)
- Focus coverage on public functions; private helpers can have lower coverage
- Do not chase 100% — diminishing returns and brittle tests
- Exclude generated code and boilerplate from coverage analysis

## Module Import Patterns

### Sampler Built Module Import

In Sampler projects, always import from the build output, not the source:

```powershell
BeforeAll {
    $moduleName = 'MyModule'

    # Remove any previously loaded version
    Get-Module $moduleName -All | Remove-Module -Force -ErrorAction SilentlyContinue

    # Import from build output
    $modulePath = Join-Path $PSScriptRoot '..' '..' 'output' 'builtModule' $moduleName
    $latestVersion = Get-ChildItem $modulePath -Directory | Sort-Object Name -Descending | Select-Object -First 1
    Import-Module (Join-Path $latestVersion.FullName "$moduleName.psd1") -Force -ErrorAction Stop
}
```

### Direct Source Import (Non-Sampler)

```powershell
BeforeAll {
    # Dot-source the function under test
    . (Join-Path $PSScriptRoot '..' 'source' 'Public' 'Get-Widget.ps1')
}
```

## Skipping Tests

### Skip Individual Tests

```powershell
It 'Should connect to Azure' -Skip:(-not $env:AZURE_SUBSCRIPTION_ID) {
    # Only runs when Azure credentials are available
}
```

### Skip by Platform

```powershell
It 'Should read the registry' -Skip:(-not $IsWindows) {
    # Windows-only test
}
```

### Pending Tests (Not Yet Implemented)

```powershell
It 'Should handle concurrent requests' -Skip {
    # TODO: Implement this test
}
```

## Output and Debugging

### Verbosity Levels

| Level | Shows |
|---|---|
| `None` | Nothing (CI-friendly, check exit code) |
| `Minimal` | Summary only |
| `Normal` | Failed tests + summary (default) |
| `Detailed` | All tests + summary |
| `Diagnostic` | Everything including discovery, mock setup, timing |

### Debug Output in Tests

```powershell
It 'Should process correctly' {
    $result = Get-Widget -Name 'Test'

    # Write-Host is visible during test runs (use for debugging)
    Write-Host "DEBUG: Result = $($result | ConvertTo-Json -Depth 3)" -ForegroundColor Yellow

    $result | Should -Not -BeNullOrEmpty
}
```

**Note**: Remove `Write-Host` debug lines before committing. Use `Write-Verbose` with `-Verbose` preference for persistent diagnostic output.

## Pester 4 to Pester 5 Migration Checklist

| Pester 4 | Pester 5 | Notes |
|---|---|---|
| `Should Be` | `Should -Be` | All operators become parameters |
| `Should BeExactly` | `Should -BeExactly` | Case-sensitive comparison |
| `Should Throw` | `Should -Throw` | Exception testing |
| `Should Not Throw` | `Should -Not -Throw` | Negated exception testing |
| `Assert-MockCalled` | `Should -Invoke` | Mock verification |
| `Assert-VerifiableMock` | `Should -InvokeVerifiable` | Verifiable mock check |
| `-TestCases` | `-ForEach` | Data-driven tests |
| Bare code in `Describe` | Move to `BeforeAll` | Discovery vs run phase |
| `$TestDrive` variable | `TestDrive:` PSDrive | Same concept, different syntax |
| `Set-TestInconclusive` | `Set-ItResult -Inconclusive` | Inconclusive results |
| `Set-TestInconclusive` | `Set-ItResult -Skipped` | Skipped results |
| `New-MockObject` | `New-MockObject -Type [T]` | Still supported |
| `Invoke-Pester -Script` | `Invoke-Pester -Configuration` | Use PesterConfiguration |
